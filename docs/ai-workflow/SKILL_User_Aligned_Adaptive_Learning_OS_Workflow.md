# 用户对齐的 Adaptive Learning OS 开发协作 Skill

版本：v1.0  
用途：让 ChatGPT、Codex 和未来新对话都按照用户已经确认的方式协作  
适用项目：

```text
/Users/issiyua/Documents/Adaptive_Learning_OS
```

核心目标：

```text
让用户始终知道：
现在在做什么
为什么要做
需要执行什么
什么算成功
什么算失败
下一步是什么
```

---

# 1. 沟通语言与表达方式

## 1.1 中文优先

默认使用中文。

出现英文术语时，必须立即解释，例如：

```text
regression fixture（回归测试样本）：
把真实出现过的 Bug 数据保存成一个可重复运行的测试输入。
以后每次修改代码，都重新跑这个测试，防止同一个 Bug 再次出现。
```

禁止直接连续使用用户可能不理解的英文词，例如：

```text
fixture
invariant
release gate
fallback
runtime artifact
stale state
scope freeze
```

正确写法：

```text
fixture（测试样本）
invariant（必须始终成立的规则）
release gate（发布前必须通过的检查清单）
fallback（主流程失败后的安全备用方案）
runtime artifact（真实运行时留下的任务记录、日志和数据）
stale state（界面仍在使用旧状态）
scope freeze（本轮只修已确认问题，不顺便加新功能）
```

## 1.2 先说结论，再说原因

回答顺序优先采用：

```text
1. 当前结论
2. 用户现在要做什么
3. 为什么
4. 详细步骤
5. 下一步
```

不要先写很长的背景，再让用户自己寻找行动项。

## 1.3 不让用户自己推断

不能只说：

```text
建议测试一下。
```

必须写成：

```text
测试文件：
选中内容：
Ask 问题：
操作步骤：
预期结果：
什么算 Pass：
什么算 Fail：
失败后需要保存什么：
```

---

# 2. 一轮对话只做一种任务

每轮只能专注下面一种：

```text
测试设计
测试结果分析
Bug Inventory（Bug 清单）
开发任务撰写
开发报告审查
QA 任务撰写
提交与阶段收尾
```

禁止在用户还在测试时，直接生成开发任务。

禁止在开发任务讨论中，顺便开始新的测试规划或新功能设计。

标准流程：

```text
完整测试
→ 汇总 Bug 清单
→ 用户确认
→ 统一开发任务
→ 开发报告
→ QA
→ 用户最小人工确认
→ 用户确认
→ commit
→ 阶段收尾
```

---

# 3. 用户手工结果优先

如果出现：

```text
自动测试全部通过
但用户在真实 Obsidian 中反复复现 Bug
```

必须采用：

```text
用户真实行为是事实
自动测试说明测试覆盖不足
```

不能说：

```text
测试没复现，所以没有 Bug。
```

应当：

```text
1. 保存用户真实输入
2. 找到对应 Ask job、日志和 Markdown
3. 转成回归测试样本
4. 修复代码
5. 再做真实 Obsidian 测试
```

---

# 4. 什么是“回归测试样本”

## 4.1 中文定义

回归测试样本，意思是：

```text
把某一次真实 Bug 的完整输入保存下来，
以后每次改代码都自动重放一次。
```

例如用户真实遇到：

```text
Ask：
测试一下，编一个小故事

AI 回答：
有完整故事

解析结果：
有 key_answer 和 suggested_takeaway

错误结果：
编辑建议 empty，Apply 不可用
```

应保存为测试样本，包括：

```text
问题
来源类型
原始 Markdown
AI 原始回答
解析结果
合并结果
预期 draft
预期 Apply 状态
```

修复以后，每次测试都验证：

```text
编辑建议不能 empty
必须生成 draft
Apply 必须可用
```

## 4.2 为什么重要

没有这种测试样本时：

```text
修完一个 Bug
以后改别的代码
同一个 Bug 可能悄悄回来
```

有了以后：

```text
每次改代码
自动检查旧 Bug 有没有复发
```

---

# 5. 四个永久机制

## 5.1 真实 Bug 必须变成自动测试

简单说：

```text
用户发现一次真实 Bug
→ 不只是修掉
→ 还要把它保存成自动测试
```

目的：

```text
防止同一个问题以后再次出现
```

---

## 5.2 阶段完成前必须过固定检查

这叫“发布前检查清单”。

不是只看：

```text
代码能编译
```

而是必须同时通过：

```text
TypeScript 检查
重点测试
全部自动测试
production build（正式构建）
重新加载 Obsidian 插件
真实 Obsidian 操作测试
marker 唯一性检查
KnowledgeData 边界检查
用户最小人工确认
```

只有这些都通过，才可以说：

```text
这个阶段完成了
```

---

## 5.3 改完代码后必须重新加载插件

插件代码写进 `main.js` 后，Obsidian 可能仍在内存中运行旧版本。

所以真实测试前必须：

```text
1. production build
2. 确认 main.js 已更新
3. Disable 插件
4. Enable 插件
```

或者：

```text
完全退出 Obsidian
重新打开
```

没有重新加载插件，真实 UI 测试结果无效。

---

## 5.4 修 Bug 时冻结范围

意思是：

```text
本轮只修已经确认的问题
不要顺便加新功能
```

例如当前只修：

```text
empty 编辑建议
重复 marker 检查
block 插入位置
```

就不要顺便做：

```text
Review Scheduler
PDF 导入
自动合并所有 block
新的 KnowledgeData 功能
```

原因：

```text
范围越大
新 Bug 越多
越难判断哪次修改造成问题
消耗的 Codex 额度也越多
```

---

# 6. 测试任务的默认格式

测试任务必须优先设计成：

```text
一份主测试笔记
顺序执行多个 Case
只有会污染数据的测试才创建副本
```

不要默认拆成十几个小文件。

每个 Case 必须包含：

```text
测试目的
测试位置
初始内容
选中内容
Ask 问题
操作步骤
预期结果
Pass 标准
Fail 标准
错误示例
需要保存的证据
结果填写模板
```

测试设计要考虑：

```text
最少切换文件
最少复制内容
最少重复操作
不同测试互不污染
破坏性测试放最后
```

---

# 7. 开发任务的默认格式

每份 DEV/FIX 文件必须包含：

```text
1. 当前问题总结
2. 用户真实测试结果
3. 项目路径
4. 真实 Vault 路径
5. 真实测试文件路径
6. 需要读取的报告
7. 需要检查的代码
8. 需要检查的 Ask job 和日志
9. 根因方向
10. 必须始终成立的规则
11. 不能使用的投机修法
12. 必须新增的自动测试
13. production build
14. 插件重新加载
15. 真实 Obsidian 测试
16. 报告格式
17. 禁止 commit
18. 完成标准
```

同时必须提供：

```text
下载链接
建议放置路径
发给 Codex 的精确提示词
报告输出路径
```

---

# 8. QA 任务的默认格式

QA 不能只重复跑自动测试。

必须区分：

```text
自动测试
真实 Obsidian 测试
用户最终人工确认
```

QA 文件要写清：

```text
要验证哪些旧 Bug
要验证哪些成功路径不能被破坏
要使用哪个测试文件
要重新加载插件
每个测试的 Pass/Fail
失败时保存 job id、截图和 Markdown
```

---

# 9. Codex 额度节省规则

详细输入，简短输出。

## 9.1 输入任务必须详细

开发任务要把用户意图、边界和测试写清楚。

不能为了省 token 而写得模糊，否则 Codex 会：

```text
猜错需求
反复修改
浪费更多额度
```

## 9.2 Codex 报告必须紧凑

报告只需要：

```text
根因
修改文件
新增测试
自动验证结果
是否重新加载插件
真实 UI 测试结果
已知限制
是否可以进入 QA
```

不要每轮更新大文档。

## 9.3 docs/context 集中更新

只在阶段最终完成时更新：

```text
docs/context
handoff
master plan
archive
```

不要每次小修都更新。

---

# 10. Commit 规则

没有用户明确同意：

```text
不要 commit
不要 tag
不要 push
```

正确流程：

```text
开发完成
→ QA
→ 用户真实确认
→ 用户说可以提交
→ 才生成 commit 任务
```

---

# 11. 阶段收尾规则

一个阶段完成时，才统一做：

```text
更新 docs/context
更新项目状态
归档 incoming/report
记录已完成能力
记录未解决问题
生成下一阶段入口
commit
```

---

# 12. 持续改进规则

这个 Skill 不是一次写完后永远不变。

每次阶段结束，都要问：

```text
这轮哪里浪费了时间？
哪里让用户看不懂？
哪里需要用户重复提醒？
测试设计是否太分散？
Codex 是否读了太多无关文件？
是否忘记 reload 插件？
是否过早生成开发任务？
```

如果发现新规律，就更新本 Skill。

版本记录示例：

```text
v1.1
- 测试默认使用单一主笔记
- 破坏性测试才使用副本

v1.2
- production build 后强制 reload 插件
- live smoke 必须记录 reload 方法

v1.3
- 英文术语必须同时给中文解释
```

---

# 13. 每次回答前的检查清单

在回复用户之前，先检查：

```text
我现在是在做测试、开发还是总结？
有没有混入另一种任务？
用户是否已经同意进入开发？
有没有使用没解释的英文术语？
有没有明确告诉用户下一步？
如果生成任务文件：
  是否有下载链接？
  是否有目标路径？
  是否有 Codex 提示词？
  是否有报告路径？
测试是否尽可能集中在一份笔记？
是否明确写了 Pass/Fail？
是否提醒 production build 后 reload 插件？
```

---

# 14. 给未来 ChatGPT / Codex 的执行指令

在处理 Adaptive Learning OS 前，请先遵循：

```text
1. 中文优先。
2. 英文术语必须解释。
3. 一轮只做一种任务。
4. 用户手工 Bug 优先于未复现的自动测试。
5. 未经用户同意，不生成开发任务。
6. 未经用户同意，不 commit。
7. 测试优先使用一份主笔记。
8. 所有任务都写清文件、操作、预期、Pass/Fail。
9. 开发后必须 build、reload plugin、真实 UI 测试。
10. 阶段完成后再统一更新 docs/context。
```
